The whole idea

A presentation is just one HTML file and three support files you never touch.

Self-contained
Open the file in a browser. That is the whole runtime.
Driven from the keyboard
Arrows, Home/End, a slide rail, full-screen, deep-link per slide.
Print to PDF
The same file exports clean, one slide per page.
You write content. The component does the deck.

Brief to ship

The build, in five moves

  • 1Brief. Audience, slot length, brand palette, deploy or local.
  • 2Scaffold. New folder; copy in the three support files.
  • 3Build index.html. Start from the example; one section per slide.
  • 4Smoke test. Serve locally, screenshot every dense slide, check for clipping.
  • 5Deploy (optional). Add the password gate, push to Vercel, verify live.
Smoke test is not optional. Render every slide before you call it done; overflow only shows in the browser.

The palette-override pattern

Re-brand by remapping tokens, never by editing CSS

/* a <style> block after the CSS links */ :root { --accent: #2A5A52; --accent-ink: #1C3F39; --slide-bg: #FAF6F1; --surface: #FFFCF6; }
Same deck, new brand
--accentbuttons, rules, chart fills
--slide-bgthe canvas behind every slide
--surfacewarm paper cards, never stark white
Charts use var(--token) fills, so they re-skin with the palette automatically.

Baked-in conventions

The rules the skill applies for you

  • Every slide passes the "what is this trying to say?" test — purpose explicit, not implied.
  • Every data series carries its named source on the slide — not just the headline one.
  • Diagram labels meet the same accuracy bar as prose — honest range, never the dramatic round number.
  • A slide of hard truths ends on agency — a "so what" card, never just the bad news.
WHY IT MATTERS
These were learned the hard way, over real feedback rounds. The skill applies them before you have to ask.
